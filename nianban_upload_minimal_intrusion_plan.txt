WeFlow 念伴上传最小侵入方案说明

1. WeFlow 当前导出链路的关键文件和关键触发点

导出页面 UI
- src/pages/ExportPage.tsx
- 导出弹窗、导出任务队列、导出成功后的状态更新都在这里。
- “导出后上传至念伴”入口也在这里，而且只对单联系人私聊显示。

导出任务创建点
- src/pages/ExportPage.tsx
- createTask() 负责把当前导出参数封装成 ExportTaskPayload，然后 enqueueExportTask(...)。
- 真正开始跑导出是在 runNextTask() 里调用 window.electronAPI.export.exportSessions(...)。

导出成功后可拿到结果路径的位置
- electron/services/exportService.ts
- exportSessions(...) 在每个会话成功导出后，把 sessionOutputPaths[sessionId] = outputPath 写回结果对象。
- src/pages/ExportPage.tsx
- runNextTask() 在 exportSessions(...) 成功后读取 result.sessionOutputPaths，并拿 targetJsonPath 作为“导出成功后触发上传”的边界输入。

导出 IPC / main / worker
- electron/preload.ts
- export.exportSessions(...) 和 export.uploadExportedChatToNianban(...) 都在这里桥接。
- electron/main.ts
- ipcMain.handle('export:exportSessions', ...) 负责把导出请求交给 export worker / main fallback。
- ipcMain.handle('export:uploadExportedChatToNianban', ...) 负责把上传请求交给独立 uploader service。
- electron/exportWorker.ts
- 只负责调用 exportService.exportSessions(...)，并把进度和最终结果回传给主进程。

导出 JSON 的落盘位置
- electron/services/exportService.ts
- exportSessions(...) 里先算 exportBaseDir、sessionDir、preferredOutputPath / outputPath。
- 真正写 JSON 的函数是 exportSessionToDetailedJson(...)。
- outputPath 即本次单会话详细 JSON 文件路径，也会写入 sessionOutputPaths。

图片 / 语音导出的落盘目录规则
- electron/services/exportService.ts
- getMediaLayout(outputPath, options) 决定媒体目录相对 JSON 的布局。
- exportImage(...) 负责图片落盘。
- exportVoice(...) 负责语音落盘。

当前代码里的稳定规则如下：
- 布局 A：JSON 在 outputDir/texts/ 下，媒体在 outputDir/images/、outputDir/voices/ 等兄弟目录；JSON 里会写 ../images/... 或 ../voices/... 相对路径。
- 布局 B（shared）：JSON 在 outputDir 下，媒体在 outputDir/media/<jsonBaseName>/images|voices 下；JSON 里会写 media/<jsonBaseName>/... 相对路径。
- 布局 C（per-session）：JSON 在会话目录下，媒体在同目录 media/images|voices 下；JSON 里会写 media/... 相对路径。

2. 哪些代码我没有动，以及为什么不动

明确不动的导出核心
- electron/services/exportService.ts 的 exportSessions(...)
- electron/services/exportService.ts 的 exportSessionToDetailedJson(...)
- electron/services/exportService.ts 的 getMediaLayout(...)
- electron/services/exportService.ts 的 exportImage(...)
- electron/services/exportService.ts 的 exportVoice(...)
- electron/exportWorker.ts

不动原因
- 这些就是 WeFlow 当前图片 / 语音 / JSON 导出的核心链路。
- 用户这次的核心要求是“先确保 WeFlow 原有导出能力不被破坏，再在导出完成后追加上传层”。
- 因此没有把念伴上传逻辑塞回 exportService，也没有改动媒体落盘逻辑、文件筛选逻辑或导出线程逻辑。

3. 上传功能接入点在哪

接入边界
- src/pages/ExportPage.tsx 的 runNextTask()。
- 只有在 window.electronAPI.export.exportSessions(...) 返回 success 且拿到 targetJsonPath 后，才异步调用 uploadExportedChatToNianban(...)。
- 如果本地导出失败，上传根本不会开始。
- 如果上传失败，只更新任务卡片里的上传状态，不回滚本地导出结果。

桥接层
- electron/main.ts
- electron/preload.ts

独立上传服务
- electron/services/nianbanUploadService.ts
- 现在上传逻辑集中在这一个独立 service 内部，不进入 export 核心实现。

4. 我如何读取导出产物

输入
- sessionId
- outputDir
- jsonPath
- 念伴上传配置（baseUrl / deviceId / dyadId）

读取方式
- 优先使用 exportSessions(...) 返回的 jsonPath。
- 如果没有显式 jsonPath，则在 outputDir 下回溯最近的详细 JSON 文件，并匹配 session.wxid。
- 只接受 WeFlow 详细 JSON 导出，不接受 arkme-json。

解析内容
- 读取 JSON 中的 session.wxid、session.type、messages[]。
- 只允许单联系人私聊；如果 session.wxid 以 @chatroom 结尾或 session.type 表示群聊，则直接拒绝上传。

媒体定位规则
- 图片：优先依赖 JSON message.content 里的相对路径，并相对 jsonPath 所在目录解析成真实文件路径。
- 语音：优先依赖 JSON message.content 里的相对路径。
- 如果语音 content 不是路径（例如开启“语音转文字”后 content 被写成转写文本），则回退扫描本次导出目录里的 voice 文件，并按导出文件名规则匹配。

当前可稳定匹配的语音规则
- WeFlow 导出的语音文件名规则来自 exportVoice(...):
  voice_<sessionToken>_<localId>_<createTime>_<serverId>.wav
- uploader 不反推原始 localId；只用 createTime + platformMessageId(serverId) 做后缀匹配。
- 如果 createTime + platformMessageId 能唯一命中一个 voice 文件，则认为可稳定匹配。
- 如果匹配不到，或者同一个键命中多个文件，则直接跳过，不硬绑。

5. 文本 / 图片 / 语音分别怎么上传

文本消息
- 从详细 JSON 的 messages[] 里筛出 localType=1 的可上传文本。
- 跳过系统消息、引用消息、link/quote app message、空文本。
- 通过 POST /dyad/:id/import/private_chat_messages 分批导入。

图片消息
- 只处理 localType=3。
- 必须先从 JSON content 解析出导出后的本地图片文件相对路径，并确认真实文件存在。
- 然后走：
  1) POST /assets/upload-session
  2) 按返回的 COS 上传说明上传文件
  3) POST /assets/complete
  4) 取回 asset_id
  5) 用带 asset_id 的结构化消息，再调用 /dyad/:id/import/private_chat_messages 写入消息

语音消息
- 只处理 localType=34。
- 优先用 JSON content 的相对路径直接定位 wav 文件。
- 如果 JSON content 里不是路径，而是“语音转文字”文本，则按上面的 voice 文件命名规则回退匹配。
- 匹配到文件后，上传链路与图片相同：
  1) /assets/upload-session
  2) COS 上传
  3) /assets/complete
  4) 拿 asset_id
  5) 用结构化媒体消息导入

统计返回
- importedCount
- textImportedCount
- imageUploadedCount
- voiceUploadedCount
- imageImportedCount
- voiceImportedCount
- skippedCount
- failedCount

6. 我认为当前最小侵入方案的风险点

风险 1：念伴资产接口的请求/响应协议在仓库里没有现成文档
- 当前实现按用户给出的链路做了独立适配层，并把 request / response 解析写得比较宽松，兼容 upload_url / headers / fields / asset_id 等常见字段。
- 如果真实后端协议字段名不同，还需要按实际接口再微调，不需要改 WeFlow 导出器。

风险 2：语音文件回连不是 100% 稳定
- 当 JSON content 是相对路径时，稳定。
- 当 JSON content 被“语音转文字”覆盖时，只能靠 createTime + platformMessageId 回退匹配导出的 wav 文件。
- 如果 platformMessageId 缺失，或者同一时间戳下存在多个 voice 文件，就会被安全跳过。
- 这是有意设计：宁可跳过，也不误绑。

风险 3：图片消息没有额外的兜底匹配
- 当前图片依赖 JSON content 的相对路径。
- 如果用户没有导出图片文件，或者 JSON content 只有 [图片] 占位符，则 uploader 会跳过，不会尝试猜测。

风险 4：媒体导入和文本导入分两阶段
- 当前先导文本，再导媒体。
- 这样可以保证媒体失败时文本仍可先入库，但前提是假设念伴按 create_time 展示消息，而不是严格按接口调用顺序。
- 如果后端严格按导入顺序展示，需要后续改成“先上传资产，再按原消息顺序混合导入”。

风险 5：全量 typecheck 在当前仓库环境不可直接作为验收信号
- 当前本地环境缺少完整 Electron / Node / 依赖类型，仓库本身跑全量 tsc 会报大量环境型错误。
- 本次改动额外做了 esbuild 解析级检查，确认新改的 TS / TSX 文件可以被成功解析和打包。

7. 测试步骤

基础文本上传
1. 打开导出页。
2. 选择一个单联系人私聊。
3. 导出格式选 JSON。
4. 打开“导出后上传至念伴”。
5. 填写 baseUrl / X-Device-Id / Dyad ID。
6. 关闭图片、语音导出，只保留文本。
7. 执行导出。
8. 确认：
   - 本地 JSON 成功生成
   - 上传状态显示 textImportedCount > 0
   - 上传失败不会影响本地文件保留

图片上传
1. 同样选择单联系人私聊。
2. 保持 JSON 格式。
3. 打开图片导出。
4. 执行导出并上传。
5. 确认：
   - 本地 images 目录有真实图片文件
   - JSON 中对应图片消息 content 是相对路径
   - 上传结果里 imageImportedCount > 0

语音上传（content 直接是路径）
1. 选择包含语音消息的私聊。
2. 打开语音导出。
3. 关闭“语音转文字”。
4. 执行导出并上传。
5. 确认：
   - 本地 voices 目录有 wav 文件
   - JSON 中语音消息 content 是相对路径
   - voiceImportedCount > 0

语音上传（content 是转写文本，走回退匹配）
1. 选择包含语音消息的私聊。
2. 同时打开语音导出和“语音转文字”。
3. 执行导出并上传。
4. 确认：
   - 本地 voices 目录仍有 wav 文件
   - JSON 中语音消息 content 可能是转写文本
   - 如果 createTime + platformMessageId 能唯一匹配 voice 文件，则 voiceImportedCount > 0
   - 如果不能唯一匹配，则 skippedCount 增加，但不会误绑到别的文件

失败隔离
1. 填一个错误的念伴地址或错误 Dyad ID。
2. 执行导出并上传。
3. 确认：
   - 本地 JSON / 图片 / 语音仍正常落盘
   - 任务卡片显示上传 error
   - 本地导出结果不受影响
